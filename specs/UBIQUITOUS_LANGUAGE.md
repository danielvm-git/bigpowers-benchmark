## Benchmark execution

| Term | Definition | Aliases to avoid |
|------|------------|-----------------|
| **Run** | The atomic unit of benchmarking: one Task × one Model × one Sandbox → one scored, persisted result | execution, job, attempt |
| **BenchRow** | The persisted JSON representation of a completed Run, stored as a shard in `~/runs/data/` | record, result, row |
| **Task** | A benchmark challenge: a scaffold codebase with bugs, features, or refactoring challenges that opencode is asked to solve | challenge, problem, exercise |
| **TaskSuite** | A named, ordered collection of Task IDs used to group Tasks for browsing and selection; a filter, not a unit of execution | suite, batch, set |
| **Sandbox** | A long-lived Daytona environment with a specific `bigpowersRef` of the bigpowers skillset pre-installed; opencode always runs inside a Sandbox, never on the host | container, environment, instance |
| **bigpowersRef** | A git ref (tag or commit SHA) identifying the version of the bigpowers skillset installed inside a Sandbox; written automatically to `BenchRow` at run time, never user-entered | version, ref, tag |
| **Workspace** | The directory inside a Sandbox where the Task repo is cloned for a single Run; fresh-cloned before each Run and deleted afterward | working directory, task dir |
| **opencode** | The AI coding agent under test; runs inside the Sandbox and produces the artifacts that Grading evaluates | agent, runner, subprocess |

## Scoring

| Term | Definition | Aliases to avoid |
|------|------------|-----------------|
| **Grading** | The fully automated process that runs `score_run.sh` inside the Sandbox after opencode exits and emits the three sub-scores | evaluation, assessment, scoring |
| **overallScore** | The single computed float (0.0–1.0) summarizing a Run: `(codePass × 2 + artifactScore + conventionScore) / 4` | score, total score, final score |
| **codePass** | Sub-score: 0 or 1 — did `test.js` exit 0? | pass/fail, test result |
| **artifactScore** | Sub-score: 0–2 — count of `.md` files in `specs/` (0 = 0, 1–2 = 1, 3+ = 2) | artifact rating, spec score |
| **conventionScore** | Sub-score: 0–2 — quality of git commit messages (0 = no commits, 1 = commits without Conventional Commits format, 2 = ≥1 Conventional Commit) | commit score, convention rating |
| **Regression** | A statistically significant drop in `overallScore` (or a sub-score) for a given Model × Task pair across consecutive `bigpowersRef` versions | decline, degradation |
| **Delta** | The signed difference in `overallScore` between two Runs or two `bigpowersRef` versions; displayed as ↑ / ↓ / → with color coding | diff, change, delta score |

## Relationships

- A **Run** belongs to exactly one **Task**, one **Model**, and one **Sandbox**.
- A **Run** maps 1:1 to a **BenchRow** on disk.
- A **TaskSuite** contains one or more **Tasks**; a **Task** may belong to multiple **TaskSuites**.
- A **Sandbox** carries exactly one **bigpowersRef**; the **BenchmarkRunner** reads that ref and stamps it on the **BenchRow**.
- A **Provider** offers one or more **Models**; a **Model** belongs to exactly one **Provider**.
- **Grading** produces exactly three sub-scores (`codePass`, `artifactScore`, `conventionScore`) from which `overallScore` is derived.
- **BenchmarkStore** persists all **BenchRows** and drives **GitOps**; **BenchmarkRunner** delegates persistence to **BenchmarkStore**.

## Example dialogue

> **Dev:** "When the user clicks Run in **Mission Control**, does the app immediately start **opencode**?"
>
> **Domain expert:** "No — the **BenchmarkRunner** first resets the **Workspace** inside the **Sandbox** by deleting and re-cloning the **Task** repo at the **bigpowersRef**. Only then does it launch **opencode**."
>
> **Dev:** "What happens if the same **Task** × **Model** combination is run twice on the same **Sandbox**?"
>
> **Domain expert:** "Two separate **BenchRows** are written with distinct UUIDs and timestamps. Neither overwrites the other — they coexist in `~/runs/data/`."
>
> **Dev:** "How does `overallScore` relate to the three sub-scores?"
>
> **Domain expert:** "`overallScore = (codePass × 2 + artifactScore + conventionScore) / 4`. **codePass** is weighted double because passing tests is the minimum bar. **Grading** is fully automated — `score_run.sh` is the only source of truth; no human entry."
>
> **Dev:** "If a **Model's** **overallScore** drops between two **bigpowersRef** versions, what do we call that?"
>
> **Domain expert:** "A **Regression**. The **Analytics** screen surfaces these automatically with a **Delta Badge** showing the signed difference."

## Flagged ambiguities

- **"score"** — used loosely to mean any of `codePass`, `artifactScore`, `conventionScore`, or `overallScore`. Prefer the specific field name. Use **overallScore** only for the computed 0.0–1.0 aggregate; use **sub-score** when referring to any of the three components collectively.
- **"run" (verb vs noun)** — "to run a benchmark" vs "a Run". Capitalize **Run** when referring to the entity to distinguish from the verb. In Swift code, use the type name `Run` / `BenchRow` consistently.
- **"skill"** — refers exclusively to a bigpowers methodology skill (e.g. `investigate-bug`, `plan-work`), never to a Model's general capability. A Model's capability category is its **Tier**. Skill Insights shows methodology skill effectiveness, not model capabilities.
- **"model"** — in SwiftUI code, "model" often means the view model or `@Observable` store. In this domain, **Model** means an LLM. Use **view model** for the SwiftUI concept to avoid collision.
- **"ref"** — appears as `bigpowersRef`, `git ref`, and informally as "version". The canonical term is **bigpowersRef** for the bigpowers skillset version; use "git ref" only in comments about the underlying git mechanism.
- **"task"** — the word appears in both "Task Library" (a **Task** entity) and in everyday speech ("the current task" meaning the user's work item). Capitalize **Task** when referring to the benchmark entity from the task repo.
