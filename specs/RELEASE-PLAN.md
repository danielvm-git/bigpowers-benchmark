## Discovery Notes

Key facts confirmed before drafting steps:

**Daytona REST API** (from `/Users/danielvm/.opensrc/repos/github.com/daytonaio/daytona/main`):
- List sandboxes: `GET https://app.daytona.io/api/sandbox` — `Authorization: Bearer <token>`
- Sandbox `labels` map stores custom metadata; `bigpowersRef` stored as `labels["bigpowers_ref"]`
- Execute async command: `POST /toolbox/{sandboxId}/toolbox/process/session/{sessionId}/exec` with `{ "command": "...", "runAsync": true }` → returns `cmdId`
- Stream logs via WebSocket: `wss://.../{sandboxId}/toolbox/process/session/{sessionId}/command/{commandId}/logs?follow=true`
- Sandbox state enum: `creating | started | stopped | error | ...` — only `state == "started"` is safe to run against
- **Security note**: `Authorization: Bearer <token>` header must never appear in logs, `print()`, or `URLError` descriptions. All `DaytonaClient` errors must strip headers before re-throwing.

**opencode CLI** (from `/Users/danielvm/.opensrc/repos/github.com/anomalyco/opencode/dev/packages/opencode/src/cli/cmd/run.ts`, line 236–240):
- Invocation: `opencode run --model <provider/modelId> --dir <taskDir> --dangerously-skip-permissions --format json @/tmp/bigpowers_prompt_<runID>.txt`
- Task prompt is written to a temp file (`/tmp/bigpowers_prompt_<runID>.txt`) inside the sandbox before invocation and deleted after grading — never interpolated into a shell command string (shell injection prevention).
- `--format json` emits newline-delimited JSON events; each line is a `LogLine` JSON object `{"t":"<ISO8601>","kind":"info|ok|warn|err|cmd","text":"..."}`.
- `--model` takes `provider/modelId` format (e.g., `openrouter/anthropic/claude-sonnet-4.6`).

**opencode exit handling**: When the log stream closes, `BenchmarkRunner` must poll the command's exit status via `GET /toolbox/{sandboxId}/toolbox/process/session/{sessionId}/command/{cmdId}` and verify exit code == 0 before proceeding to grading. A non-zero exit yields `.failed(RunnerError.opencodeNonZeroExit(code:))`.

**models.dev API** (confirmed from `https://models.dev/api.json`):
- No auth required; returns flat JSON keyed by provider ID, each with nested `models` object
- Fields per model: `name`, `reasoning` (bool), `inputModalities`, `outputModalities`, capability flags

**score_run.sh output** (from `DOMAIN.md`):
```json
{ "code_pass": 0, "artifact_score": 0, "convention_score": 0, "token_cost": 0.0 }
```
All four keys are required. Defensive decode: treat missing/null keys as 0. `score_run.sh` may not be present in all sandboxes — check exit code; yield `.failed(RunnerError.gradingScriptMissing)` if command fails with "not found".

**Architecture decisions**:
- DOMAIN.md (authoritative) requires opencode to run inside a Daytona sandbox. This plan follows DOMAIN.md.
- One Daytona session is created per full Run and reused across all phases (workspace reset, opencode, grading). Sessions are not created per-command.
- Command construction (building argument lists for Daytona exec calls) happens exclusively inside `BenchmarkRunner` actor — never in View or ViewModel code.
- `~/runs/data/` must be a pre-existing git repo. The app checks this on launch and shows an onboarding sheet if it is absent or not a git repo.

## Red-Flag Check

Rationalizations caught during planning:

1. **Pure Package.swift vs Xcode project** — Risk: Keychain, Notifications, and IOPMAssertion require entitlements for distribution. For a single-developer tool, they work without entitlements. A minimal `.entitlements` file is added in Story 1.1 so the app runs consistently from the first `swift run`. Documented as Risk #1 below.
2. **`swift build` as UI verify** — All UI steps use `swift build` since there is no CLI-runnable UI test. This is the minimum bar. The Verification Script covers manual checks. Accessibility steps are included per-story, not deferred.
3. **DaytonaClient abstraction** — Justified as testability requirement; see Story 3.1 Reason for Depth.
4. **Shell injection via prompt** — Caught and mitigated: prompt written to temp file, never string-interpolated into commands.
5. **TDD order** — Steps in all stories are ordered test-first: failing test → production code → refactor.

| 1 | Keychain + Notifications + IOPMAssertion without `.entitlements` may fail when app is signed for distribution | `BigPowersBenchmark.entitlements` created in Story 1.1; document required capabilities |
| 2 | `URLSessionWebSocketTask` log streaming holds an actor-isolated reference — possible retain cycle | Explicitly cancel WebSocket task in timeout path of `BenchmarkRunner` actor |
| 3 | Daytona API shape may diverge from the local source snapshot | `MockDaytonaClient` insulates unit tests; integration test planned post-v1.0 |
| 4 | `models.dev/api.json` field names may change | `ModelRegistry` uses optional decode + graceful nil; re-fetches when cache expires |
| 5 | `score_run.sh` output format not yet in repo | Fixture test in Story 3.2b validates expected schema; missing keys default to 0 |
| 6 | Swift Package Manager `.executableTarget` + SwiftUI `@main` conflict with test setups | All testable logic in `BigPowersBenchmarkKit` library; executable contains only `@main` + scenes |
| 7 | `Sandbox.state` may gain new values in future Daytona releases | `SandboxState` enum has `.unknown` fallback; `isRunnable` predicate is the single filter point |
| 8 | Task prompt containing shell metacharacters could inject commands | Prompt written to temp file in sandbox; never interpolated into command string |
| 9 | `git push` remote may require HTTPS with embedded token | Document that SSH remote or system credential helper is required; never construct URLs with embedded tokens |
